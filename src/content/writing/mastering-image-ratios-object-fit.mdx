## The Silent Killer of Web Performance

Images are patient. They'll wait until the last possible moment to load, and when they do, they'll take exactly as much space as their natural dimensions require — regardless of what's already on the page. For users, that means content jumping. For Core Web Vitals, it means a CLS hit. For you, it means a client bug ticket.

Two lines of CSS fix it. The demo below shows exactly what's happening and why.

## The Problem

This is the problem we're trying to solve	with Object-Fit. It's quite prevalent on older sites, or ones that haven't yet adopted effective responsive design practices. By refreshing the page, if it wasn't noticeable on page load, the example below - our 'bad behavior' demo - will highlight exactly the issue that Object-Fit solves. These images have none of these practices, nor size attributes which means that not only do we have a decreased Cumulative Layout Shift (CLS) score, but a poor user experience. The card content will appear first at the top of the row, until the image loads, where it will then shift down to its expected position. Pretty nasty, right? Couple this with a slow internet connection, and it can become a real nightmare, and fast.

## How object-fit works

`object-fit` controls the relationship between an image's natural dimensions and the space its container has reserved for it. Without it, the browser makes that decision for you — and it almost always chooses wrong for UI components like cards, headers, and profile images.

The property accepts five values. The starter guide covers all of them in detail — the one that matters for CLS and consistent card layouts is cover.

```css
img {
  width: 100%;
  height: 100%;
  object-fit: cover;
}
```
cover fills the container completely, cropping the image if the aspect ratios don't match. The image is never distorted, never letterboxed — it just fits, cleanly, whatever the container dimensions are. That's the fix the demo above is using.

Its companion property is object-position, which controls which part of the image stays visible when cover crops it. The default is center — which works for most images — but for a portrait photo where the subject's face is at the top, object-position: top prevents the crop from cutting them out.

```css
img {
  width: 100%;
  height: 100%;
  object-fit: cover;
  object-position: top;
}
```

But `object-fit` alone doesn't prevent layout shift. It controls how the image fills its container — it doesn't tell the browser how big that container should be before the image loads.

## Two properties, not one

`object-fit` controls how an image fills its container. It doesn't control how big that container is before the image loads — and that's the gap that causes layout shift.

Without a fixed container size, the browser doesn't know how much vertical space to reserve for an image slot. It allocates nothing, renders the surrounding content, and then reshuffles everything when the image arrives. That's CLS — and object-fit alone can't prevent it.
The fix is aspect-ratio on the container:

```css
.card-img-wrap {
  aspect-ratio: 16 / 9;
  overflow: hidden;
}

.card-img-wrap img {
  width: 100%;
  height: 100%;
  object-fit: cover;
  object-position: center;
}
```

`aspect-ratio` tells the browser exactly how tall the slot will be before any image arrives. Space is reserved instantly. When the image loads, it has a container waiting for it — nothing on the page moves.

You might have seen the old approach to this: the padding hack, where a percentage padding-top on a container with position: relative faked a fixed ratio. It worked, but it was fragile, unintuitive, and a nightmare to explain to the next developer. aspect-ratio replaced it entirely and has been Baseline since 2021.

## The performance upside

Once the container has a fixed ratio, you know its exact display dimensions at every breakpoint. That's not just useful for layout stability — it means you can serve precisely sized images.

Without a fixed ratio, you're guessing. You serve a large image and hope the browser crops it acceptably. With `aspect-ratio: 16 / 9` on a container that's 600px wide, you know the display size is 600×338px. You can use srcset to serve an image that matches those dimensions exactly, instead of downloading a 1200px original and letting `object-fit: cover` handle the crop in the browser.

```html
<div class="card-img-wrap">
  <picture>
    <source
      srcset="image-600x338.jpg 600w, image-1200x675.jpg 1200w"
      sizes="(max-width: 768px) 100vw, 600px"
    />
    <img
      src="image-600x338.jpg"
      alt="Descriptive alt text"
      width="600"
      height="338"
      loading="lazy"
    />
  </picture>
</div>
```
The width and height attributes on the `<img>` element reinforce the aspect ratio as a hint to the browser before the CSS loads — a belt-and-braces approach that's worth the two extra attributes.
	
The result: smaller file transfers, no layout shift, and consistent cropping across every card in the grid. That's what the CLS demo at the top of this post is showing — not just a visual fix, but a performance strategy.

## Go fix your images

If you've got a card grid in production, open DevTools and run a Lighthouse audit. A CLS score above 0.1 on image-heavy pages is almost always this — unsized images loading into unsized containers. Two properties, five minutes, done.

`aspect-ratio` reserves the space. `object-fit: cover` fills it cleanly. Add `width`, `height`, and `loading="lazy"` while you're there. That's the whole pattern — no library, no build step, no polyfill needed.
